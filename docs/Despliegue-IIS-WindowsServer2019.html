2 Requisitos previos del servidor

Ejecutar todos los comandos en PowerShell como Administrador en el servidor de destino (Windows Server 2019).

2.1 Instalar Node.js LTS

Descargar el instalador MSI desde https://nodejs.org/en/download (versión LTS ≥ 20.x) e instalar con opciones por defecto.

# Verificar instalación
node --version   # → v20.x.x o superior
npm  --version   # → 10.x o superior

2.2 Instalar IIS con los módulos necesarios

Install-WindowsFeature -Name Web-Server, Web-Common-Http, Web-Static-Content, `
  Web-Default-Doc, Web-Http-Errors, Web-Asp-Net45, `
  Web-Http-Redirect, Web-Health, Web-Http-Logging, `
  Web-Security, Web-Filtering, Web-Mgmt-Console `
  -IncludeManagementTools

2.3 Instalar URL Rewrite Module

Descargar desde https://www.iis.net/downloads/microsoft/url-rewrite y ejecutar el instalador MSI.

# Verificar instalación desde PowerShell
Get-WebConfiguration system.webServer/rewrite/rules -ErrorAction SilentlyContinue
# No debe lanzar error si está instalado

2.4 Instalar Application Request Routing (ARR)

Descargar desde https://www.iis.net/downloads/microsoft/application-request-routing e instalar el MSI.

# Habilitar el proxy global de ARR (ejecutar UNA sola vez)
Import-Module WebAdministration
Set-WebConfigurationProperty -pspath 'MACHINE/WEBROOT/APPHOST' `
  -filter "system.webServer/proxy" -name "enabled" -value "True"

2.5 Instalar PM2 (gestor de procesos Node.js)

npm install -g pm2
npm install -g pm2-windows-startup
pm2-startup install    # Registra PM2 como servicio de Windows

2.6 Instalar Angular CLI (solo para compilar)

npm install -g @angular/cli

2.7 Puertos y Firewall

3 Preparación de Active Directory

3.1 Obtener los datos del dominio

Ejecutar en un Domain Controller o máquina con RSAT instalado:

$domain = Get-ADDomain

# LDAP_URL — FQDN del Domain Controller principal
"LDAP_URL=ldap://" + (Get-ADDomainController).HostName

# LDAP_BASE_DN — Distinguished Name del dominio
"LDAP_BASE_DN=" + $domain.DistinguishedName

# AD_DOMAIN — Nombre NetBIOS (ej. LAB-MH)
"AD_DOMAIN=" + $domain.NetBIOSName

3.2 Crear la cuenta de servicio

Esta cuenta es usada por Node.js para buscar usuarios en AD. Solo necesita permisos de lectura (Domain Users es suficiente).

$pass = Read-Host -AsSecureString "Contraseña para svc-rdweb"

New-ADUser `
  -Name              "svc-rdweb" `
  -SamAccountName    "svc-rdweb" `
  -UserPrincipalName "svc-rdweb@lab-mh.local" `
  -AccountPassword   $pass `
  -Enabled           $true `
  -PasswordNeverExpires $true `
  -Description      "Cuenta de servicio RDWeb-Moderno (solo lectura)"

# Verificar
Get-ADUser svc-rdweb | Select-Object Name, UserPrincipalName, Enabled

3.3 Obtener datos del RD Connection Broker y Gateway

# RDCB_SERVER
Get-RDServer -Role RDS-CONNECTION-BROKER | Select-Object Server

# RDGATEWAY_HOSTNAME
Get-RDServer -Role RDS-GATEWAY | Select-Object Server

4 Despliegue del backend Node.js

4.1 Copiar archivos al servidor

Copiar la carpeta backend\ del proyecto al servidor:

# Ruta destino en el servidor
C:\inetpub\wwwroot\rdweb-moderno\backend\
ℹ️
Nota No copiar la carpeta node_modules\ — se instala en el servidor.
# En el servidor, instalar dependencias
cd C:\inetpub\wwwroot\rdweb-moderno\backend
npm install --production

4.2 Crear el archivo .env

Crear el archivo .env en C:\inetpub\wwwroot\rdweb-moderno\backend\.env con el siguiente contenido:

# ── Servidor ──────────────────────────────
PORT=3000
NODE_ENV=production

# ── JWT (generar con el comando de abajo) ─
JWT_SECRET=REEMPLAZAR_CON_SECRETO_GENERADO
JWT_EXPIRES_IN=8h

# ── Active Directory ──────────────────────
LDAP_URL=ldap://DC01.lab-mh.local
LDAP_BASE_DN=DC=lab-mh,DC=local
AD_DOMAIN=LAB-MH
AD_SERVICE_USER=svc-rdweb@lab-mh.local
AD_SERVICE_PASS=CONTRASEÑA_DE_SVC_RDWEB

# ── RDS ───────────────────────────────────
RDCB_SERVER=SRV-APP.LAB-MH.LOCAL
RDGATEWAY_HOSTNAME=rdgateway.lab-mh.local

# ── Simulación (desactivada en producción) 
SIMULATION_MODE=false
# Generar un JWT_SECRET seguro con PowerShell
-join ((1..32) | ForEach-Object { '{0:x2}' -f (Get-Random -Max 256) })
🔐
Seguridad El archivo .env contiene credenciales. Asignar permisos NTFS restrictivos:
Solo la cuenta de servicio de PM2 / sistema local debe poder leerlo.

4.3 Crear ecosystem.config.js

Este archivo es la configuración de PM2. Crear en C:\inetpub\wwwroot\rdweb-moderno\backend\ecosystem.config.js:

module.exports = {
  apps: [{
    name:        'rdweb-backend',
    script:      'src/index.js',
    cwd:         'C:\\inetpub\\wwwroot\\rdweb-moderno\\backend',
    instances:   1,
    watch:       false,
    env_production: {
      NODE_ENV: 'production'
    }
  }]
};

4.4 Registrar el backend con PM2

cd C:\inetpub\wwwroot\rdweb-moderno\backend

# Iniciar el backend
pm2 start ecosystem.config.js --env production

# Guardar la lista de procesos (para que reinicie solo)
pm2 save

# Verificar que está corriendo
pm2 status
pm2 logs rdweb-backend --lines 20
# Verificar que el backend responde
Invoke-RestMethod http://localhost:3000/api/health
# Debe retornar: { status: "ok", simulationMode: false, ... }

5 Compilar y desplegar el frontend Angular

💡
La compilación se puede hacer en cualquier máquina con Node.js y Angular CLI. El resultado (archivos estáticos) se copia al servidor.

5.1 Configurar el environment de producción

En la máquina de desarrollo, editar frontend\src\environments\environment.prod.ts:

export const environment = {
  production: true,
  apiUrl: '/api',  // Relativo — IIS hace el reverse proxy
};

5.2 Compilar para producción

cd RDWeb-Moderno\frontend

# Compilar en modo producción
ng build --configuration production

# Los archivos compilados quedan en:
#   frontend\dist\frontend\browser\

5.3 Copiar al servidor

# Copiar el contenido de dist\frontend\browser\ al servidor
C:\inetpub\wwwroot\rdweb-moderno\frontend\   ← destino

La estructura final en el servidor debe quedar:

C:\inetpub\wwwroot\rdweb-moderno\
  ├── frontend\          ← Archivos Angular compilados
  │   ├── index.html
  │   ├── main-HASH.js
  │   ├── styles-HASH.css
  │   └── web.config     ← (creado en el paso 6)
  └── backend\
      ├── src\
      ├── node_modules\
      ├── .env
      └── ecosystem.config.js

6 Configuración de IIS

6.1 Crear el sitio web en IIS

Abrir IIS Manager

Ejecutar inetmgr en PowerShell o desde el menú Inicio → Herramientas administrativas → Internet Information Services.

Crear nuevo sitio

Click derecho en Sites → Add Website. Completar:
· Site name: rdweb-moderno
· Physical path: C:\inetpub\wwwroot\rdweb-moderno\frontend
· Binding: HTTPS, puerto 443 (o HTTP:80 para pruebas iniciales).
· Hostname: el FQDN que usarán los usuarios (ej: rdweb.lab-mh.local)

Application Pool

El pool por defecto No Managed Code es correcto para servir archivos estáticos de Angular. Verificar que esté en No Managed Code (no ASP.NET).

6.2 Crear el web.config para Angular + Reverse Proxy

Crear el archivo C:\inetpub\wwwroot\rdweb-moderno\frontend\web.config:

<?xml version="1.0" encoding="UTF-8"?>
<configuration>
  <system.webServer>
    <rewrite>
      <rules>

        <!-- REGLA 1: /api/* → Node.js en puerto 3000 (Reverse Proxy) -->
        <rule name="API Reverse Proxy" stopProcessing="true">
          <match url="^api/(.*)" />
          <action type="Rewrite"
                  url="http://localhost:3000/api/{R:1}" />
        </rule>

        <!-- REGLA 2: Angular SPA — todas las rutas → index.html -->
        <rule name="Angular SPA" stopProcessing="true">
          <match url=".*" />
          <conditions logicalGrouping="MatchAll">
            <add input="{REQUEST_FILENAME}"
                 matchType="IsFile"
                 negate="true" />
            <add input="{REQUEST_FILENAME}"
                 matchType="IsDirectory"
                 negate="true" />
          </conditions>
          <action type="Rewrite" url="/index.html" />
        </rule>

      </rules>
    </rewrite>

    <!-- MIME types adicionales -->
    <staticContent>
      <mimeMap fileExtension=".webmanifest"
               mimeType="application/manifest+json" />
    </staticContent>

    <!-- Cache para assets con hash (Angular los versiona) -->
    <httpProtocol>
      <customHeaders>
        <add name="X-Content-Type-Options" value="nosniff" />
        <add name="X-Frame-Options"        value="SAMEORIGIN" />
      </customHeaders>
    </httpProtocol>

  </system.webServer>
</configuration>

6.3 Habilitar ARR Proxy (si no se hizo antes)

Import-Module WebAdministration
Set-WebConfigurationProperty `
  -pspath  'MACHINE/WEBROOT/APPHOST' `
  -filter  "system.webServer/proxy" `
  -name    "enabled" `
  -value   "True"

# Reiniciar IIS para aplicar cambios
iisreset

6.4 Certificado SSL (recomendado para producción)

Si tienes una CA interna (ADCS) en el dominio, solicitar un certificado desde Administrador de IIS → Server Certificates → Create Domain Certificate. Alternativa: usar Let's Encrypt con win-acme.

7 Verificación final

Verificar Backend directo 
Invoke-RestMethod http://localhost:3000/api/health

Esperado: { "status": "ok", "simulationMode": false }

Verificar IIS sirviendo el Frontend

Abrir en navegador: http://rdweb.lab-mh.local — debe aparecer la pantalla de login.

Verificar Reverse Proxy 
Invoke-RestMethod http://rdweb.lab-mh.local/api/health

Si responde igual que el paso 1, el proxy está funcionando.

Verificar Login con AD real

Ingresar con usuario del dominio. Si falla, revisar los logs de PM2:

pm2 logs rdweb-backend --lines 50
Verificar Cookie en DevTools

F12 → Network → login request → Response Headers debe mostrar set-cookie: rdweb_token=.... Luego en Application → Cookies → el sitio debe mostrar rdweb_token.

Probar lanzamiento de una app

Hacer click en una RemoteApp del dashboard. Debe descargarse un archivo .rdp. Abrirlo debe iniciar la conexión a través del RD Gateway.

8 Resolución de problemas comunes

SíntomaCausa probableSolución
Login falla con "Credenciales incorrectas" AD_SERVICE_USER/PASS incorrecto o cuenta bloqueada Verificar con Test-ADAuthentication
set-cookie no aparece en DevTools CORS bloqueando la cookie cross-origin en dev Usar proxy de Angular en dev (proxy.conf.json) — en prod IIS es same-origin
Dashboard queda en "Cargando..." infinito Sin zone.js, Angular no detecta cambios async ChangeDetectorRef.detectChanges() implementado — reiniciar si persiste
502 Bad Gateway en /api/* Node.js no está corriendo (PM2 caído) pm2 restart rdweb-backend
404 en rutas de Angular (ej: /dashboard al recargar) URL Rewrite no redirige a index.html Verificar que el web.config tiene la regla "Angular SPA"
Cannot connect to LDAP Firewall bloqueando puerto 389 hacia el DC Test-NetConnection dc01.lab-mh.local -Port 389
HTTP 401 en /api/apps después de login Cookie no se envía en la petición Verificar withCredentials: true o que el interceptor esté registrado en app.config.ts

9 Actualizar la aplicación

Actualizar solo el backend

# Copiar nuevos archivos src\ al servidor, luego:
cd C:\inetpub\wwwroot\rdweb-moderno\backend
npm install --production   # si hay nuevas dependencias
pm2 reload rdweb-backend      # zero-downtime restart
pm2 logs rdweb-backend --lines 20  # verificar sin errores

Actualizar solo el frontend

# En máquina de desarrollo:
ng build --configuration production

# Copiar dist\frontend\browser\ al servidor
# IMPORTANTE: mantener el web.config existente
# IIS sirve los nuevos archivos automáticamente — no requiere reinicio

Actualizar variables de entorno (.env)

# Editar C:\inetpub\wwwroot\rdweb-moderno\backend\.env
# Luego reiniciar el proceso (el .env se carga al inicio):
pm2 stop    rdweb-backend
pm2 start   ecosystem.config.js --env production
pm2 save

Lista de verificación para una instalación nueva

  • Node.js LTS instalado y verificado (node --version)
  • IIS instalado con los feature roles necesarios
  • URL Rewrite Module instalado
  • Application Request Routing (ARR) instalado y habilitado
  • PM2 instalado globalmente y configurado como servicio de Windows
  • Cuenta de servicio svc-rdweb creada en AD
  • Archivo .env creado con todos los valores reales
  • Backend corriendo en PM2 (pm2 status = online)
  • Frontend compilado y copiado a frontend\
  • Sitio web en IIS creado apuntando a la carpeta frontend\
  • web.config con reglas de Angular SPA y API Proxy
  • ARR proxy habilitado globalmente
  • Test de login exitoso con usuario del dominio
  • Archivo .rdp descargado y conexión establecida vía Gateway
